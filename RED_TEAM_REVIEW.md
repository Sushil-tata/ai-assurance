# Red-Team Review (Part 13)

**Audience:** sponsors and reviewers who want the honest critique, not the sales pitch.
**Method:** each item below states the risk plainly, then either accepts it with a stated reason, mitigates it with a specific change already reflected elsewhere in this repository, or flags it as a genuine open trade-off.

## 1. Unnecessary tables
- `fact_human_approval` versus `fact_committee_decision` — two tables that could arguably be one with a `decision_tier` column. **Kept separate** because analyst pre-review and committee decision have different required fields (rationale length, delegated authority checks) and merging them would force nullable columns exactly like the pattern ADR-009 argues against elsewhere. Judged worth the duplication.
- `fact_finding_contributing_factor` versus just repeating `root_cause_code` as an array on `fact_finding` — the bridge table is arguably over-engineered for MVP's single scenario, which has exactly one finding with three factors. **Accepted as-is**: this is cheap normalization that will matter the moment a second product/model generates findings with different factor combinations, and the marginal cost now is near zero.

## 2. Unnecessary agents
- Seven agents for one MVP scenario is arguably more agents than the demo strictly needs — a simpler build could merge Data Quality into Monitoring, or merge Assurance/Critic into Investigation with a "two-pass" prompt. **We kept them separate deliberately**: the entire credibility argument of this system rests on the critic being a *structurally independent* reviewer (see `agents/ASSURANCE_AGENT.md` §"Business purpose"). A merged agent reviewing its own output is a materially weaker governance story, not just a simpler one. This is the one place we will not simplify for build speed.
- **Genuine open question, not resolved here:** whether Data Quality Agent needs to be fully separate from Monitoring Agent for MVP specifically, versus being folded in with a narrower tool boundary. We kept them separate to match the brief's explicit seven-agent requirement and because Scenario 001's bureau-missingness cause benefits from an agent whose only job is data health. A build under severe time pressure could justify merging these two for MVP only, with a documented plan to split them again before enterprise scale — this is a legitimate simplification if Step 7 of `DELIVERY_PLAN.md` is running over budget.

## 3. Data duplication
- `fact_account_snapshot_monthly.delinquency_bucket_id` duplicates information that also lives in `fact_delinquency_state_monthly.bucket_current`. **Accepted**: the snapshot-level pointer supports fast filtering/joins for C Score eligibility without requiring every query to join the full delinquency-state table; the delinquency-state table remains the source of truth for the richer collections detail. Documented explicitly in `TABLE_CATALOG.md` Domain 8 rather than left as a silent redundancy.
- `fact_application.a_score_value` denormalizes a value that also lives in `fact_model_score_event`. **Accepted with a documented caveat** in the table spec: `fact_model_score_event` is canonical; the application-table copy is convenience-only. Risk: if these ever drift, which one wins? **This is a real gap** — no explicit reconciliation rule is defined. Recommended fix before enterprise build: either drop the denormalized field or add a nightly consistency check (a new DQ rule).

## 4. Ambiguous grain
- `fact_evidence_item.source_row_identifier` is a free-text string (`"table.column=value"` convention), not a strongly typed FK, because evidence can point into tables across two different platforms (Fabric Warehouse and Dataverse) with no single foreign-key mechanism spanning both. **This is a genuine weakness**: the convention is enforced by discipline and code review, not by the database. Mitigation: `testing/DATA_TEST_CASES.md` should include a check that every `source_row_identifier` string actually resolves (a runtime validation, not a schema constraint) — **not currently in the test suite, flagged as a gap to add**.

## 5. Temporal leakage risks
- The biggest real risk in this design is a feature engineering team building `dim_feature_definition` rows with an incorrect `observation_lag_days` (e.g., accidentally computing "3-month average utilization" using a window that extends past `observation_date`). The schema *supports* catching this (DQ-002), but DQ-002's enforcement depends entirely on `fact_feature_lineage.source_row_reference` being populated accurately — **if the feature pipeline doesn't populate lineage correctly, the leakage check is toothless**. This is a build-discipline risk, not a schema flaw, but it's the single highest-consequence failure mode in the whole system and deserves explicit sign-off attention during Step 5 of `DELIVERY_PLAN.md`.

## 6. Immature label risks
- ADR-006 and the `maturity_status` filter are sound in design. The residual risk is **human override of the filter under deadline pressure** ("just show me what we have, mark it draft") — the schema cannot prevent a person from manually querying immature labels outside the governed pipeline. Mitigation: this is a training/discipline issue, addressed in `policies/MODEL_MONITORING_POLICY.md` §4.1 as a policy statement, but **not technically enforced against ad hoc query access** — a genuine, accepted limitation of any data platform, not unique to this design, but worth stating plainly rather than implying the schema alone prevents misuse.

## 7. Synthetic data credibility risks
- Any synthetic dataset risks looking "too clean" — real bureau data has more chaotic missingness patterns, real customer behaviour has more outliers than a generator naturally produces. **Accepted risk for MVP**: `data/SYNTHETIC_DATA_SPEC.md` explicitly calls for realistic noise and autocorrelation, but a reviewer with real portfolio experience may still find the demo data "too tidy." Mitigation: the demo script (`demo/JUDGE_NARRATIVE.md` §4) explicitly tells presenters not to oversell data realism — be upfront that this is illustrative, not a real portfolio.

## 8. Over-engineering
- The Governance Agent's Azure AI Search index is arguably more infrastructure than a 6-clause MVP policy library needs — a hardcoded lookup table could serve the same demo. **Kept as designed** because it is the one component that meaningfully demonstrates grounded RAG done correctly (citation-verified, index-versioned) rather than a toy hardcoded answer, which matters for the "not architecture theatre" requirement and for showing the RAG pattern done safely.
- The full 45-table schema for a demo that only populates ~15 of those tables with live data could be seen as over-built for MVP. **Accepted deliberately** — see `ADR-001` and `MVP_VS_ENTERPRISE.md`: the entire point of this repository is proving the schema generalizes, and that claim is worthless without the full schema actually existing, reviewed, and specified to implementation depth, even where unpopulated.

## 9. Copilot Studio limitations
- Multi-step autonomous reasoning (hypothesis generation → evidence gathering → counter-evidence search → synthesis) is more naturally expressed in a code-based agent framework with explicit state machines than in Copilot Studio's primarily conversational topic model. **Genuine risk, flagged in `DELIVERY_PLAN.md` Step 7**: the Investigation Agent may need more explicit prompt scaffolding (numbered required steps, not just described responsibilities) than a first draft of `prompts/INVESTIGATION_INSTRUCTIONS.md` provides, and iteration time should be budgeted, not assumed away.
- Copilot Studio's connected-agent hand-off model can make deeply nested multi-agent chains (Orchestrator → Investigation → Governance → back to Investigation → Assurance) harder to debug than a single-process call stack. Mitigated by the correlation-ID tracing design (`architecture/OBSERVABILITY.md`), but debugging experience will still be slower than a code-first framework — an accepted trade-off for the native Teams/Dataverse/Entra integration benefits (ADR-002).

## 10. RAG hallucination risks
- The Governance Agent's policy search is the one place in the system where retrieval quality directly determines answer quality — a poorly chunked or embedded policy document could cause it to retrieve an irrelevant clause with a confident-sounding citation. Mitigated by: relevance thresholding (agent reports "not found" below a configured relevance score, per `agents/GOVERNANCE_AGENT.md` §"Confidence handling") and by every citation being independently checkable (document/version/clause number). **Residual risk**: relevance thresholding still requires tuning against real query patterns, which won't be known until the MVP is actually used — flagged as a post-launch tuning task, not a solved problem.

## 11. Unsupported policy interpretation
- The Governance Agent is explicitly constrained to answer only from literal clause text and refuse interpretation beyond it (`prompts/GOVERNANCE_INSTRUCTIONS.md` §11). The risk this creates: it may be *unhelpfully* literal on genuinely ambiguous policy questions, frustrating users who want a synthesized answer. **Accepted trade-off**: an overly helpful, interpretive Governance Agent is a bigger risk (silently making policy calls it has no authority to make) than an overly literal one. Documented explicitly so it's a known UX trade-off, not an oversight.

## 12. Agent recursion or routing failure
- The critic-rejection loop has an explicit circuit breaker (two rejections → escalate), preventing infinite recursion. **Checked, sound.**
- The Orchestrator has no equivalent circuit breaker for a hypothetical routing loop (e.g., if two specialist agents each believed the other should handle a request). **Genuine gap**: no test case in `testing/AGENT_TEST_CASES.md` currently exercises a routing ping-pong scenario. Recommended addition before enterprise build: an explicit hop-count limit on Orchestrator hand-offs (e.g., max 3 hops per user turn) with a hard stop and human-facing "unable to complete this request" message.

## 13. Weak human accountability
- The design is deliberately strict here (ADR-005) and this is judged to be sound, not a weakness. The one soft spot: `fact_human_approval`'s `approval_stage = analyst_review` is optional (recommendations can skip straight to committee) — meaning a low-severity finding could reach committee without any analyst-level sanity check first, adding to committee workload. **Accepted trade-off**: better to over-include the committee than to create an analyst-level bypass path that could be misused to avoid committee scrutiny on a borderline case.

## 14. Unrealistic MVP scope
- The `DELIVERY_PLAN.md` 9.5-week estimate assumes one person **already proficient** in Copilot Studio, Fabric, and credit risk model monitoring domain knowledge simultaneously. **This is optimistic.** A more realistic estimate for a single builder new to any one of these three areas is 12–14 weeks. This should be stated to sponsors plainly rather than let the clean step-by-step plan imply more certainty than it has.
- The seven-agent, six-tool MVP is at the upper edge of what's genuinely minimal — a stripped-down "prove the pattern" version could plausibly ship with five agents (merging Data Quality into Monitoring, per §2 above) and four tools in significantly less time, at the cost of a slightly less complete governance story. **This is the single biggest lever available if the timeline needs to compress.**

## Final reduced recommendation

Keep the full enterprise-grade schema (Parts 2–5) and the full governance/evidence model (Part 6) exactly as designed — these are cheap to specify fully now and expensive to retrofit later, and they are what makes the MVP demo credible to a model risk audience rather than a toy. If the timeline is under real pressure, the two legitimate simplification levers are: **(a)** merge Data Quality Agent into Monitoring Agent for MVP only, with a documented plan to re-split before enterprise scale, and **(b)** reduce the six regression scenarios to Scenario 001 plus one failure-path scenario (recommend keeping Scenario 005, MOB eligibility, since it's the brief's single most emphasized hard constraint) for the initial ship, adding the remaining four post-launch. Do not simplify the critic loop, the human-approval gate, or the deterministic-metric separation — those three are the actual thesis of this project, and removing any of them would produce a different, weaker system, not a leaner version of this one.
